Трирівнева модель сховища

Коли проєкти накопичують знання незалежно, команди переучують одні й ті ж уроки щоразу. Трирівнева модель це виправляє: Ваш агент зберігає знання, упорядковані за доменами, у своєму сховищі. Спільне сховище проєкту містить наскрізні домовленості. Підключіть командне сховище, щоб ділитися між проєктами.

Пріоритет пошуку: сховище агента → сховище проєкту → командне сховище.

1 Сховище агента — фронтенд, бекенд, наскрізні домени
↓
2 Сховище проєкту — командні домовленості, рішення
↓
3 Командне сховище — спільне для всіх проєктів
# Підключити командне сховище $ soleri vault connect git@github.com:team/design-vault.git ✓ Підключено командне сховище 142 entries ✓ Об’єднано індекс пошуку ✓ Пріоритет: агент → проєкт → команда
# Підвищити патерн до командного сховища $ soleri vault promote pattern-semantic-tokens \ --target team ✓ Підвищено до командного сховища ✓ Доступно всім учасникам команди

Патерни накопичуються між проєктами

# Пов’язати споріднені проєкти $ soleri vault link /project-a /project-b \ --type related ✓ Проєкти пов’язано (двонапрямно) ✓ Увімкнено міжпроєктний пошук
# agent.yaml — конфігурація сховища vault: backends: - type: local path: ~/.soleri/vaults/my-agent - type: git uri: git@github.com:team/vault.git sync: on-start push: on-promote

Пов’язуйте проєкти, діліться патернами

Пов’язуйте проєкти як споріднені, батьківський/дочірній або форк. Коли ви шукаєте у своєму сховищі, пов’язані проєкти включаються автоматично із зваженою релевантністю.

Патерни, засвоєні у проєкті A, доступні у проєкті B.

Кодифікуйте робочі процеси вашої команди

Патерни кажуть, що робити. Плейбуки кажуть, як.

Плейбуки — це валідовані багатокрокові процедури, що зберігаються у вашому сховищі. Вони фіксують повний робочий процес — не лише правило, а й послідовність, очікувані результати та критерії валідації.

Приклади: міграція токенів у кодовій базі, налаштування нового компонента з нуля, налагодження збою контрастності. Кожен крок містить, що перевірити і коли ви завершили.

Повторювані процеси, а не знання в головах.

# Перелік доступних плейбуків $ soleri vault list-playbooks --category design Знайдено 4 плейбуки: design:component-setup 7 кроків design:token-migration 5 кроків design:contrast-audit 4 кроки design:theme-scaffolding 6 кроків
# Структура плейбука id: design:component-setup type: playbook steps: - name: Створити каркас файлів компонента run: soleri create component $name validate: файли існують, експорти присутні - name: Застосувати систему токенів run: soleri validate --fix validate: token score >= 95 - name: Перевірити доступність run: soleri check-contrast validate: усе PASS
# Запустити плейбук $ soleri vault run-playbook design:component-setup \ --name UserCard Крок 1/7: Створити каркас файлів компонента ✓ Крок 2/7: Застосувати систему токенів ✓ Крок 3/7: Перевірити доступність ✓ ... ще 4 кроки

# Open Source Release — Three-Layer Architecture

## Context

Key rule: personas are thin, engine is thick. A persona is just config + knowledge + optional custom ops. All heavy lifting is in the engine. When adding a feature, ask: would any other persona benefit? Yes = engine. Design-specific only? = domain module. Salvador-only? = custom op.

## The Three Layers

**Layer 1: ENGINE** — vault, brain, planning, memory, session capture, intent routing, facade loader. Distributed as npm package. All personas share it. Runs as a SINGLE MCP server process — one entry in ~/.claude.json, forever. Personas are runtime context, not separate processes. When Salvador is active and you call `soleri_vault op:search`, it searches Salvador's vault. Switch to Gaudi, same call searches Gaudi's vault. Tool names don't change — the context does.

**Layer 2: DOMAINS** — pluggable domain modules (design ops, security ops, architecture ops, testing ops). Distributed as npm packages. Personas select which domains to activate. Engine dynamically loads/unloads domain modules based on the active persona.

**Layer 3: PERSONA** — thin config layer: persona.yaml (identity, voice, domains, intent rules) + vault/ (knowledge) + optional custom ops. Generated by forge. The CLAUDE.md injection is minimal — just a table of available personas and activation phrases. On activation, the engine returns full persona context (facade tables, intent rules, voice) dynamically. Adding a new persona via forge does NOT touch CLAUDE.md.

## Single MCP Server Architecture

One engine process registers generic ops:

```
soleri_core     → activate, health, search, identity
soleri_vault    → capture, search, query
soleri_brain    → stats, build_intelligence
soleri_planning → create, approve, update, complete
soleri_memory   → search, capture, session_capture
```

The `activate` op switches persona context at runtime. No separate MCP entries per persona. 10 agents = still 1 process.

## Transport Abstraction Layer

Engine core has ZERO protocol dependencies. Pure TypeScript logic:

```
engine/
├── core/           # Pure logic. No MCP imports.
│   ├── vault.ts    # VaultService class
│   ├── brain.ts    # BrainService class
│   ├── planning.ts # PlanningService class
│   └── memory.ts   # MemoryService class
├── transports/     # Protocol adapters
│   ├── mcp.ts      # Claude Code (primary, ships with v1)
│   ├── rest.ts     # REST API (future: web dashboards, custom integrations)
│   └── lsp.ts      # LSP (future: VS Code, Cursor, Zed native extensions)
```

Adding a new transport = new adapter file wrapping core services. Core never changes. This prevents Claude Code lock-in — if Anthropic changes MCP, or if users want Cursor/Windsurf support, only the transport adapter changes.

## Vault Backend Abstraction

The vault system uses an abstract backend interface:

```typescript
interface VaultBackend {
  read(id: string): Entry
  write(entry: Entry): void
  search(query: string): Entry[]
  list(filter?: Filter): Entry[]
  migrate(fromVersion: number, toVersion: number): void
}

class LocalVaultBackend implements VaultBackend   { /* filesystem — default */ }
class GitVaultBackend implements VaultBackend      { /* git repo — team sharing */ }
class RemoteVaultBackend implements VaultBackend   { /* API sync — hosted teams */ }
```

v1 ships with LocalVaultBackend. Git and Remote backends enable team features without engine rewrites. Backend is configured per-vault, so a persona can have a local personal vault AND a remote team vault simultaneously.

## Two-Tier Vault Model (Cross-Persona Knowledge)

Personas are NOT siloed. Knowledge lives in two tiers:

```
project/
├── shared-vault/              # Project-level, all personas read/write
│   ├── patterns/              # Cross-domain patterns
│   └── sessions/              # Session memories (persist across switches)
└── personas/
    ├── salvador/
    │   └── vault/             # Salvador-specific design knowledge
    └── gaudi/
        └── vault/             # Gaudi-specific architecture knowledge
```

**Search order:** persona vault first → shared vault second. This means domain-specific knowledge takes priority but cross-domain knowledge is always accessible.

**Capture rules:** captures go to persona vault by default. User can promote to shared vault: `soleri_vault op:promote`. Session memories always go to shared vault so context survives persona switches.

**Persona switching:** "Hola, Salvador!" → engine saves current session state to shared vault → loads Salvador context → Salvador can see what Gaudi just did via shared session memory. No context loss.

## Example

```
~/.claude.json:
  "soleri": { "command": "node", "args": ["engine/server.js"] }

Engine reads forge registry → knows which personas exist

"Hola, Salvador!" → engine activates Salvador → loads design domain
  → vault search: Salvador's vault first, then shared vault
  → captures a component pattern → goes to Salvador's vault

"Hola, Gaudi!" → engine saves session to shared vault → activates Gaudi
  → loads architecture domain → vault search: Gaudi's vault first, then shared vault
  → Gaudi can see Salvador's session context via shared vault
  → Gaudi captures architecture decision → goes to Gaudi's vault

soleri_vault op:promote id:component-pattern
  → moves from Salvador's vault to shared vault
  → now visible to ALL personas
```

## Why

This separation means engine improvements automatically benefit all personas. Domain improvements benefit all personas using that domain. No duplication, no manual propagation. Single MCP server means no resource waste, no config bloat. Transport abstraction prevents platform lock-in. Vault backend abstraction makes team features additive, not a rewrite. Two-tier vault model means personas share context without losing domain specificity.
